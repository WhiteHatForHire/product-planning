# V5.2 — Data Export + Memory Search Lite

> Imported from the source archive and converted to Markdown for searchability. Treat the source path as provenance, not as the current canonical location.
V5.2 — Data Export + Memory Search Lite

Version 0.1 — Locked decisions Status: READY TO BUILD after V5.1 Council revisions are merged Ships as two separate PRs: Data Export first, Memory Search Lite second.

Goal

Give users full ownership of their data and let them retrieve relevant past moments from chat without vector RAG complexity. Trust and transparency feature. Required for user data sovereignty.

Build sequence

PR 1: Data Export (lower safety complexity, contained Settings/API work, easy QA) PR 2: Memory Search Lite (chat integration, safety framing, redaction behavior)

Both are V5.2. They ship as separate PRs, not combined.

PR 1 — Data Export

Three export formats

JSON — full raw dump wrapped in a clean export envelope.

{

"exported_at": "...",

"format_version": "v1",

"user_id": "...",

"tables": {

"stable_profile": [],

"check_ins": [],

"commitments": [],

"practice_progress": [],

"practice_choices": [],

"practice_scenario_feedback": [],

"event_log": [],

"saved_syntheses": []

}

}

Field names stay close to DB/API reality for technical portability. Do not over-normalize.

CSV — one file per table, zipped.

anchor-export-YYYY-MM-DD.zip

profile.csv

check_ins.csv

commitments.csv

practice_progress.csv

practice_choices.csv

practice_scenario_feedback.csv

event_log.csv

saved_syntheses.csv

ZIP library: agent picks standard Node approach (archiver is acceptable).

Markdown — human-readable recovery summary in Anchor voice. Not clinical, not therapeutic. Tone: "Here is a clear overview of the recovery work this person has recorded in Anchor."

Structure:

Export overview: export date, user identifier, date range covered, note: "This export was generated from Anchor data."

Recovery focus: substance/behavior focus, start date, current day count, recovery approach if available

Commitments: completed, active, repeated themes — no over-interpretation

Check-in trends: averages and simple counts only, no giant raw dump

Practice work: modules completed, repeated practices, last practiced dates

Notable history: selected event_log entries, relapse/slip support events, milestone events, high-signal saved synthesis paragraphs

AI-generated saved reflections: only paragraphs the user opted to save, clearly labeled

Footer: "This is not a medical record or clinical assessment. Anchor may summarize patterns, but does not predict relapse."

AI-generated content labeling across all formats:

Markdown: "> Generated by Anchor AI, saved by user on [date]"

CSV: generated_by,Anchor AI and saved_by_user,true

JSON: "generated_by": "Anchor AI" and "saved_by_user": true

Privacy rules

Phone numbers excluded from all exports by default. User can opt in via a toggle.

Toggle label: "Include phone numbers in export" Default: off Helper copy: "Phone numbers are excluded by default. Turn this on only if you want saved contact numbers included in your export."

Saved synthesis paragraphs: included only if user opted to save them, labeled as AI-generated.

Endpoint

Single endpoint with format parameter:

GET /api/export?format=json

GET /api/export?format=markdown

GET /api/export?format=csv_zip

Endpoint group lives at /api/export.

Event logging

Event type: export_generated

Summary examples:

"User exported data in JSON format."

"User exported data in Markdown format."

"User exported data in CSV ZIP format."

No user content. No row details. No phone numbers. No filenames with sensitive info.

Settings UI

Location: Settings → Data → Export

Dedicated "Data" section inside Settings. Same general area as account deletion, but visually separate.

Data section structure:

Export your data (export button + format options + phone number toggle)

Delete account (separate subsection, not adjacent to export button)

Do not put delete next to export. Deletion feels available but not accidentally clickable.

File naming

anchor-export-YYYY-MM-DD.zip

anchor-export-YYYY-MM-DD.json

anchor-export-YYYY-MM-DD.md

Browser handles duplicates if multiple same-day exports.

Loading state

Simple loading indicator:

Standard: "Preparing your export. This may take a moment."

Large history (1000+ entries): "This account has a larger history, so export may take a little longer."

Generate synchronously for V5.2 unless the agent finds a real timeout risk during implementation. If timeout is a genuine risk, note in BUILD_REPORT as deferred streaming path.

Council of Models

Not required for Data Export. No AI-touching content. Mechanical API + UI work.

app_settings

check_ins

No new tables. export_generated is a new event_log type string (additive).

PR 2 — Memory Search Lite

What it does

Searches event_log by keyword when a user asks something memory-shaped in chat. Results are summarized into the chat response. Invoked only on explicit memory intent — never auto-injected into normal chat.

Intent detection

Rule-based keyword/phrase heuristic. No AI inference. Conservative — miss is better than irrelevant injection.

Trigger logic:

Fire when message matches Group A AND Group B, OR matches an exact trigger phrase.

Group A (time/pattern words):

before, last time, happened before, pattern, history, previous, past, again, usually, always

Group B (subject words):

felt, helped, did, happened, check-in, relapse, urge, mood, pattern, situation

Explicit trigger phrases (match any):

"has this happened before"

"when did this happen last"

"what did I do last time"

"what worked before"

"what helped before"

"what usually happens when"

"is this a pattern"

"do I always do this"

"have I said this before"

"have I been here before"

"remind me what helped"

"look back"

"from my history"

"based on my past"

"in previous check-ins"

"last time I felt this"

"last time I was like this"

"what changed since last time"

Conservative threshold: if uncertain, do not trigger. False positives feel creepy and can distort the current chat.

Search scope

Event_log first and only for V5.2.

Event_log may include entries summarized from check-ins, commitments, practice_progress, relapse_response events, milestone events, and saved syntheses. This keeps privacy and complexity contained. Do not directly search every source table in V5.2.

Date range parsing

Minimal support only. Parse obvious phrases if cheap:

"in March", "last week", "last month", "this month", "since April"

Do not overbuild natural-language date parsing. If uncertain, ignore date filter and search normally. Recency weighting is acceptable internally, but relevance wins over recency.

Result count

Default: 3 results. Hard cap: 5.

Redaction rules

Excluded from results (never injected into chat):

Full free-text check-in notes

Phone numbers

Contact details

Email addresses

Addresses

Raw journal-like entries

Crisis-contact numbers

Private saved messages intended for SOS/contact

Full AI syntheses unless specifically saved and flagged as relevant

Allowed in results:

event_log summary string

event type

timestamp/date

non-sensitive check-in metrics

high-level tags/fields

saved synthesis title/short summary if user saved it

Presentation in chat

The AI incorporates the pattern naturally in its response, then lists sourced events at the end in a small labeled section.

Format:

A few possibly relevant moments from your history:

- Mar 12: [event summary]

- Apr 03: [event summary]

- May 09: [event summary]

Do not put a history block before the AI response.

Framing lines (verbatim)

Full framing: "A few moments from your history came up as possibly relevant. They may or may not match what is happening today."

Short UI label: "Possibly relevant from your history:"

Never say: "Your history shows...", "This matches a pattern...", "You always do this when...", "Based on your history, you are likely to...", or any definitive pattern claim.

Backend helper

function getRelevantPastEvents(

query: string,

filters: { kind?: string; dateRange?: [Date, Date] },

limit: number

): Promise<EventLogEntry[]>

Scoring: keyword match in summary (primary), kind match boost, recency boost. No AI used in scoring.

Memory screen UI

Defer. No search box in Memory screen for V5.2. Chat-triggered only. Memory screen search is V5.4 or later after the behavior proves useful in production.

Schema

No new tables. Memory Search Lite is computed on demand from existing event_log data.

Council of Models

Required before merge for Memory Search Lite.

Light Council pass on:

Trigger detection copy (the "possibly relevant" framing)

Redaction rules completeness

No prediction language in framing

No definitive pattern claims

No raw notes or phone numbers in injected context

How AI references past events in chat response

Council prompt goes in the BUILD_REPORT per standard pattern.

Phase E: unit tests (trigger detection, result limits, redaction assertions)

Phase F: BUILD_REPORT + PR

Council required before merge. Auto-merge: no.

Vector RAG or embeddings

AI-powered search scoring

Memory screen search UI (V5.4)

Cross-session memory synthesis

Any modification to stable_profile

Export sharing via generated link (download-only)

Real-time streaming export (V2 if needed)
