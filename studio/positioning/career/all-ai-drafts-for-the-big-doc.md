Architect vs. Engineer

The engineer asks: how do I implement this correctly? The architect asks: what should exist, what should not exist yet, how should it be divided, what can be reversible later, and where are the failure points? Maxwell still needs enough engineering literacy to spot nonsense, but his premium comes from architectural judgment. He must know what tools and patterns are for, when to reach for them, and what tradeoffs they imply.

The AI-native operating model

Tool | Best use | When Maxwell reaches for it | Failure mode if overused

v0.dev | High-speed UI exploration and app shell drafts | When converting product ideas into concrete interface options | Mistaking generated UI for finished product design

The vocabulary of possible

Maxwell does not need to know how every framework works internally. He needs a vocabulary of possible moves. That means knowing that Supabase can remove a month of backend plumbing, that Stripe gives him billing and subscription primitives fast, that Playwright can validate end-to-end flows, that PostHog gives him product analytics from day one, that Resend can handle transactional mail, and that GitHub Actions plus Vercel can turn good repo discipline into a calm deployment pipeline.

Feature vs. product vs. platform

• A feature is a local capability. It solves one user action or one workflow moment.

• A product is a coherent loop. It has an entry point, a core action, a repeat-use reason, and a measurable outcome.

• A platform is a stable base that enables multiple products, integrations, or customer-specific workflows.

Technical risk is about what can break, scale poorly, or trap future decisions. Product risk is about whether anyone cares, pays, activates, or returns. Maxwell must learn to de-risk product assumptions before he invests heavily in technical polish.

Part 6 — The AI-Native SDLC Protocol

This is Maxwell’s repeatable solo operating system for taking a product from idea to live deployment without drowning in prompt chaos or architecture drift.

Phase 1 — Idea Capture

Input: raw idea, pain point, market observation, founder request

Output: one-paragraph concept note

Time: 30–60 minutes

Tools: notes, Claude chat

Goal: capture the idea in plain language

Gate: can explain the product in one paragraph without jargon

Phase 2 — Project Brief

Input: concept note

Output: one-page product brief with user, problem, core loop, risks

Time: 1–2 hours

Tools: Claude chat, markdown

Goal: define what product is being built and why

Gate: one clear user, one clear pain, one clear core action

Phase 3 — Technical Specification

Input: product brief

Output: system requirements, stack decision, entities, routes, constraints

Time: 2–4 hours

Tools: Claude chat, docs, diagrams

Goal: turn product intent into buildable technical language

Gate: enough specificity that AI can scaffold without inventing product logic

Phase 4 — Architecture and System Design

Input: technical spec

Output: data model, system diagram, component map, ADRs

Time: 2–4 hours

Tools: Mermaid, Excalidraw, Claude

Goal: create structural clarity before code

Gate: data flow, auth flow, and boundaries are visible and understandable

Phase 5 — Dev Environment Setup

Input: architecture package

Output: initialized repo, environments, CI baseline, project scaffolding

Time: 2–3 hours

Tools: Claude Code, Cursor, GitHub, Vercel, Supabase

Goal: create a calm build environment

Gate: repo runs locally, deploys preview, and env vars are documented

Phase 6 — Feature Development

Input: scoped implementation slices

Output: working features in sequence

Time: variable

Tools: Claude Code, Cursor

Goal: build in bounded slices

Gate: each slice has definition of done, test notes, and no unresolved architectural confusion

Phase 7 — Testing

Input: working features

Output: smoke coverage, targeted tests, manual QA notes

Time: 1–3 hours per milestone

Tools: Playwright, TypeScript, manual testing

Goal: verify critical flows before staging

Gate: auth, core loop, and billing-critical flows pass

Phase 8 — Code Review

Input: tested feature slice

Output: cleaned code, reduced duplication, type safety audit

Time: 30–90 minutes per slice

Tools: Claude, Cursor

Goal: remove obvious rot before merge

Gate: code matches the spec and no mystery code remains

Phase 9 — Staging Validation

Input: release candidate

Output: staging sign-off

Time: 1–2 hours

Tools: Vercel preview, staging DB, smoke checklist

Goal: test in near-production conditions

Gate: smoke checklist passes and monitoring is armed

Phase 10 — Production Deployment and Launch Loop

Input: staged release

Output: production deploy, launch notes, post-launch monitoring

Time: deploy day + first week loop

Tools: Vercel, Sentry, PostHog, support channels

Goal: launch, observe, learn, iterate

Gate: product is live, key events monitored, rollback path known

Protocol rules

• Never build before the one-page brief exists.

• Never let AI infer core product logic from vibes.

• Never add a new tool without a concrete problem.

• Every phase produces an artifact. If there is no artifact, the phase did not happen.

Key Takeaways

• The SDLC protocol is what protects Maxwell from chaos as much as it accelerates build speed.

• Each phase has a gate. Gates prevent sludge.

• The point is not ceremony. The point is coherence.

Part 9 — The Testing Playbook

Testing is not about proving perfection. It is about preventing stupid, expensive, trust-killing failures in the flows that matter most.

What Maxwell owes himself:

• enough confidence to deploy without dread

What he owes users:

• that auth works, core actions work, and data is not casually lost

What he owes clients:

• that critical flows were intentionally tested and release risk was not guessed at

At MVP stage, test:

• auth

• core action

• billing if present

• key form submissions

• critical integrations

• basic mobile responsiveness

Skip:

• exhaustive unit coverage on trivial UI

• elaborate integration matrices nobody will maintain

At production-credible stage, add:

• end-to-end smoke coverage

• stronger monitoring

• performance checks

• failure-path testing

• rollback rehearsal

The four layers

1. Type safety

Use TypeScript strict mode. Treat it as the first test layer.

Catches: undefined pathways, weak contracts, bad assumptions between components and services.

2. Unit testing

Use for logic that can actually break: formatters, pricing calculations, workflow transforms, permission logic.

Avoid testing trivial rendering details.

3. Integration testing

Use when multiple pieces must cooperate correctly: auth plus DB, webhook plus billing, form plus persistence.

4. End-to-end testing

Use Playwright for the flows that matter most:

• sign up / login

• onboarding

• core action

• billing

• file upload if relevant

• admin path if business critical

Smoke test checklist

• app loads in production

• login works

• onboarding or primary first-use flow works

• core object can be created

• core object can be edited

• billing path works if present

• transactional email fires if relevant

• analytics events appear

• no critical Sentry spikes after deploy

Exploratory testing

• behave like a distracted user

• behave like a maliciously impatient user

• break forms

• refresh at weird times

• use back button

• try on mobile

• try with missing required data

• trigger the edges AI rarely thinks about

Minimum browser/device standard

• latest Chrome

• Safari mobile-sized viewport

• one real phone check before important launch

Post-launch monitoring

• Sentry alerting active

• PostHog core events visible

• watch signup, activation, conversion, and crash/error spikes in first 24 hours

• know the threshold for rollback versus watchful waiting

Key Takeaways

• Test the expensive failures, not your ego.

• Type safety is part of testing.

• For Maxwell’s use case, Playwright smoke coverage is disproportionately valuable.

1. Default Stack Decision Framework

Use this one-page framework when choosing the stack for a new product. Start with the default: Next.js, TypeScript, Tailwind, Vercel, Supabase, Stripe, Resend, PostHog, Sentry, Playwright. Only depart from it when one of these is true: mobile is central now, the product is integration-first, the product is data-infrastructure heavy, or the buyer has a real enterprise constraint.

• If the product is API-first, keep the web shell light and give the API clean contracts and docs.

• If the product is mobile-first, prove the loop on web or a prototype before going deep on mobile.

• If the product is data-heavy, separate transactional concerns from analytics concerns earlier.

2. fCTO Discovery Call and Scoping Framework

A discovery call is not for proving intelligence. It is for reducing ambiguity and deciding whether the engagement deserves to exist.

• Questions to ask: What business result are you trying to create? What user pain is most urgent? What already exists? What has been tried? What technical constraints are real vs assumed? What is the timeline pressure? What is the budget reality? Who makes decisions?

• Listen for red flags: no clear user, no clear buyer, hidden redesign disguised as MVP, investors dictating fantasy timelines, founders wanting certainty without discovery, requests for unlimited scope under one price.

• Close the call with three things only: what Maxwell understands, what remains unknown, and the proposed next artifact or paid discovery step.

• Never leave the first call with an engineering commitment bigger than a scoped follow-up deliverable.

3. AI Prompt Conventions and Session Protocol

The point of prompt conventions is continuity. The model should enter every session inside the same project reality.

• Open every serious session with: product summary, current branch, current milestone, current constraints, today’s single goal, files most relevant, and what must not be changed.

• For implementation prompts, always specify: artifact, scope boundary, output format, error-handling expectation, and definition of done.

• After every major model interaction, capture the decision: what changed, why, and what future sessions must remember.

• Do not let long conversational context become the source of truth. The spec and repo should remain the source of truth.

4. Investor-Facing Technical Summary Template

This is the one-page narrative a founder can put in front of investors or advisors when the question becomes: is the technology real, credible, and sane?

• Sections: product purpose, current technical architecture, why this stack was chosen, current reliability and security posture, main technical risks, 6-month roadmap, and why the system should not require a major rewrite soon.

• Keep the voice calm and non-defensive. Investors are not asking for perfection. They are asking whether the team understands the system and its risks.

• Include one diagram, one roadmap, and one paragraph on how technical choices map to business milestones.

• Never oversell scale readiness. Explain what is ready now, what will need reinforcement later, and why the path is reasonable.
