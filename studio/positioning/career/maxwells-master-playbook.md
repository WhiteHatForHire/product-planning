PART 1 — IDENTITY AND POSITIONING

The Reframe

The engineering market is bifurcating. On one side are syntax executors whose value was historically tied to producing correct code quickly. AI coding tools are commoditizing that value at pace. On the other side are systems-thinkers who can direct AI toward working software, architect products, evaluate output quality, manage product scope, and ship end-to-end. Maxwell sits firmly in the second category, and that position is appreciating.

The Director Model

The most useful frame: the Director Model. In filmmaking, the director does not operate the camera. They define vision, orchestrate specialists, maintain coherence, and hold quality standards. AI-native building operates the same way. Claude Code, Cursor, and v0.dev are the specialist operators. Maxwell’s role is to define what to build, how it should be structured, what constitutes acceptable output, and when something is ready to ship.

The Governing Principle:

Every phase has a gate. The gate is binary: is the output complete enough that the next phase will not fail because of it? If yes, proceed. If no, finish the phase. The AI tools collapse execution time by 10x. The phases still exist. The thinking still has to happen.

Phase 0: Idea Capture

Input: A thought, a problem, a gap you noticed.

Output: A single-page Idea Brief containing: problem statement (one sentence), who has this problem and how often, current workaround, core value exchange, riskiest assumption, what’s out of scope.

Gate: If you can’t write the problem statement in one sentence, you don’t understand the idea yet. Do not proceed.

Phase 1: Project Brief

Input: Idea Brief. Output: 2–4 page Project Brief.

The brief must contain: product vision (one paragraph), target user persona (named, specific), core user journey (numbered happy path only), MVP feature list (Must Have / Should Have / Won’t Have), success metrics (numbers, not vanity), risk register (likelihood / impact / mitigation).

Gate: Show the brief to one person who is your target user. If they say “yes, I would use that” — proceed.

Phase 2: Technical Specification

Input: Approved Brief. Output: 5–10 page Technical Spec.

The spec must contain: stack decision with justification for each layer, complete data model (all entities, fields, types, relationships), API route design (every endpoint with method/inputs/outputs/auth/side effects), auth flow (sequence diagram), third-party integrations, environment architecture (local/staging/prod), full folder structure, non-functional requirements, and a numbered list of open questions.

Gate: A developer who has never heard of this project could read the spec and start building without asking a single clarifying question.

Phase 3: Architecture and System Design

Output: System architecture diagram, data flow diagrams for 2–3 complex journeys, infrastructure diagram, and Architecture Decision Records (ADRs).

ADR format: Status, Context, Decision, Consequences, Alternatives Considered. ADRs live in /docs and become the technical memory of the project.

Gate: Could a new technical hire understand the entire system in 30 minutes using only these diagrams and ADRs?

Phase 4: Dev Environment Setup

Repo init (private GitHub, branch protection on main). Scaffold project structure exactly as defined in spec. Environment variables (.env.local + .env.example). First Supabase migration from data model. ESLint + Prettier + Husky pre-commit hooks. CI/CD: GitHub → Vercel (dev gets preview, staging deploys to staging, main deploys to production).

Gate: npm run dev starts with no errors. DB schema exists. All env vars documented.

Phase 5: Feature Development

Every feature gets its own branch off dev. Never build two features on the same branch. The quality of your Claude Code prompt is the difference between working code and spaghetti.

Feature completion checklist: Happy path works end-to-end, loading/error/empty states implemented, mobile responsive, no console errors, TypeScript compiles clean, no hardcoded strings, RLS policies tested.

Phase 6: Testing Protocol

Layer 1 — Type Safety: TypeScript strict mode. tsc --noEmit in pre-commit hook. Non-negotiable.

Layer 2 — Unit Tests: Only for: anything handling money, auth/permissions, complex data transformations, functions with multiple code paths.

Layer 3 — E2E Tests: Playwright for: user signup/login, the core feature, payment flow, anything that if broken causes user loss or revenue loss.

Phase 7: Code Review Protocol

Paste diff into Claude. Review for: security vulnerabilities, performance issues, race conditions, missing error handling, TypeScript violations, pattern deviations. Any security flag is must-fix. Squash merge into dev.

Phase 8: Staging Validation

Merge dev into staging. Smoke test: signup, login/logout, core feature, navigation, mobile view, error states, payment (Stripe test mode), emails, data persistence. Lighthouse targets: Performance > 85, Accessibility > 90, Best Practices > 90, SEO > 90. Security check: wrong-user access, network tab for leaked data, exposed env vars.

Phase 9: Production Deployment

Pre-deploy: all env vars set in Vercel prod, production Supabase has correct schema, Stripe in live mode, custom domain + SSL, error monitoring (Sentry) active, analytics (PostHog) active. Deploy: merge staging → main → push. Post-deploy: validate within 10 minutes. If broken, revert immediately. Never fix production directly.

PART 8 — HOW MAXWELL WILL NOT FAIL

Twenty failure modes to monitor and avoid explicitly.

Build Anti-Patterns

The Context Collapse: Letting chat history get so long the AI forgets your architecture. Rule: Start new chats for new features.

Architecture Theater: Using Kubernetes for a 5-user MVP. Rule: If Vercel can’t host it, you’re over-engineering.

Skipping the Spec: Coding before the data model is final. Rule: No code until the DB schema is locked.

The Polish Trap: 4 hours on a button hover state. Rule: Use v0.dev and move on.

The Monolith Trap: Stuffing everything into one Next.js app when a service boundary makes sense. Rule: Know when an app wants a clean API split.

Tutorial Treadmill: Learning framework internals instead of building products. Rule: Architect-level knowledge only. Capabilities, limitations, cost.

Career Anti-Patterns

Commodity Drift: Taking undifferentiated work at low rates to fill the calendar. Each low-rate engagement makes the next one more likely. Set and hold a rate floor.

Portfolio Paralysis: Spending months perfecting portfolio pieces instead of launching and selling. Three shipped projects beats ten hypothetical perfect ones.

Idea Hopping: Starting new product ideas without exhausting the current one. Each new idea resets the clock. Set a minimum 6-week commitment before pivoting.

Underselling Positioning: Reverting to “developer” framing under pressure. The positioning statement is an asset. Protect it.

Single-Client Dependency: One client at 60%+ of income recreates employment without benefits. Maintain pipeline even when busy.

Identity Drift: Centering identity around web development instead of venture creation and technical leadership with AI-native execution.

Operational Anti-Patterns

No Gate Checks: Proceeding to the next SDLC phase when the current one is incomplete. The spec saves you 10x the time it costs.

No Documentation: Shipping without ADRs. Future you (or your client) will have zero context on why decisions were made.

Skipping Staging: Deploying straight to production. Always validate on staging first.

Ignoring Analytics: Launching without PostHog. If you can’t see where users drop off, you can’t improve retention.

The AI Blind Trust: Accepting generated code without reviewing architecture, security, and edge cases.

Environment Variable Exposure: Hardcoding secrets. .env.local from day one, .env.example committed, secrets never in code.

PART 9 — TESTING AND DEPLOYMENT RUNBOOK

The 80/20 of Testing

Type Safety (100% coverage): TypeScript strict mode. This eliminates an entire class of bugs.

E2E Testing (Critical Paths): Playwright for: 1) Login, 2) Payment, 3) Core Loop.

Manual QA: The “Bali Beach Test” — can a user use this on a slow 3G connection?

What you are NOT doing at MVP: 100% test coverage, load testing, chaos engineering, full accessibility audit. Those are post-launch.

The Deployment Runbook

Branch Strategy

main: Production. Protected. No direct pushes.

staging: Mirrors production. Used for QA before every deploy.

dev: Working branch. Preview URLs on every push.

Feature branches: feat/feature-name off dev. One feature per branch.

Rule: Every PR must have a Vercel Preview URL and an AI-generated summary of changes.

Pre-Deploy Checklist

All environment variables set in Vercel production environment

Production Supabase project has correct schema (migrations run)

Stripe is in live mode (not test) in production env

Custom domain configured and SSL active

Error monitoring active (Sentry or equivalent)

Analytics active (PostHog — set up before launch so day one data is captured)

Staging smoke test passed within last 24 hours

Post-Deploy Validation (Within 10 Minutes)

Production URL loads correctly

Sign up flow works on production

Core feature works on production

Payments work in live mode (real $1 test transaction)

Error monitoring dashboard shows zero new errors

Analytics events are being tracked

If broken: Revert immediately. git revert HEAD && git push origin main. Fix on dev, re-test on staging, re-deploy.

The core operating unit for AI-native building:

PART 14 — RED LINES AND RISK MANAGEMENT

Financial Red Lines

Rate Floor: Never take work below $150/hr equivalent (or $15k project minimum). Commodity work attracts more commodity work.

Single-Client Cap: No one client above 60% of income. Maintain pipeline even when busy.

Runway Rule: If runway drops below 3 months, switch to freelance-primary immediately. Product experiments require financial breathing room.

Operational Red Lines

No Code Without Spec: Phase 2 output must exist before Phase 4 begins. No exceptions.

No Production Without Staging: Every deployment goes through staging validation first.

No Launch Without Analytics: PostHog must be configured before go-live.

No Feature Without Branch: Every feature gets its own branch. Never build two features on the same branch.

Positioning Red Lines

Never say “developer”: You are a fractional CTO, venture builder, or AI-native product engineer. Protect the positioning.

Never lead with frameworks: On discovery calls, talk Product Risk, Time-to-Market, and Cost of Delay.

6-week minimum commitment: To any product experiment before pivoting. Each new idea resets the clock.

The Compounding Advantage of Speed

The structural reason this entire playbook works:

Higher margin per project: Same deliverable, fewer hours invested.

More experiments per year: For indie building, more shots on goal.

Client perception premium: Fast delivery without quality sacrifice signals mastery.

Risk reduction for clients: Rapid prototyping before full investment de-risks product bets.

The Meta-Skill

You are not trying to become a current frontend developer. You are trying to become a solo software studio. The modern stack is just the substrate. Your real craft is: product taste, scope judgment, architecture selection, AI orchestration, and launch discipline. That is the game.
