PROJECT 11 — The B2B SaaS Foundation

Why it matters for fCTO positioning: Almost every NYC startup you’ll work with is B2B SaaS. This is the architecture pattern they all need and most founding teams can’t design correctly from day one.

What you’re building:

A multi-tenant SaaS application with organizations, workspaces, and team billing. Think: a founder signs up, creates an org, invites their team, pays per seat, has an admin panel. This is Notion’s model, Linear’s model, every B2B tool’s model.

What this unlocks:

You can walk into any founding team and say “I’ve built multi-tenant from scratch, I know where it breaks, and I can set it up right the first time.” That sentence alone is worth thousands in fCTO fees because most early-stage companies get this wrong and pay for it at Series A.

What you need to know (architect level):

∙	Multi-tenancy means one codebase serves many isolated organizations — row-level security in Supabase handles the data isolation layer.

∙	Seat-based billing in Stripe means the subscription quantity updates as users are added/removed — there’s a specific webhook pattern for this.

∙	Role-based access control (RBAC): owner, admin, member, viewer — this is a table in your DB, not a hardcoded check.

∙	Organization-scoped everything: every record in your DB belongs to an org_id — this is a schema decision you make on day one or pay for later.

∙	Audit logs: enterprise clients will ask for them — every user action writes to an events table with timestamp, user, and change.

What you don’t need to know:

How to implement Casbin, how Kubernetes namespacing works, or how to build identity federation.

The fCTO signal: You can set up the technical foundation that scales from 1 to 1,000 enterprise clients without a rewrite.

Milestone: A working multi-tenant app where two separate organizations have completely isolated data, team management, and billing. Screenshot the org switcher.

PROJECT 12 — The API Product

Why it matters for fCTO positioning: Enterprise and fintech companies in NYC live and die by integrations. If your client ever wants to sell to a bank, a media company, or a Series B+ startup, they will be asked “do you have an API?” This is also its own product category — and a recurring revenue model.

What you’re building:

A public-facing API with authentication (API keys), rate limiting, versioning, and auto-generated documentation. Build it around something real — a data enrichment API, a document processing API, an AI inference API. The product is the API itself.

What this unlocks:

You understand API-first architecture. You can design and ship the layer that enterprise clients plug into. This is a completely different skill signal than “I built a web app.”

What you need to know (architect level):

∙	API keys are just long random strings stored hashed in your database — the client sends them in the Authorization header.

∙	Rate limiting lives in Redis or Upstash — count requests per key per minute, return 429 when exceeded.

∙	API versioning (/v1/, /v2/) is a routing decision that saves you from breaking clients when you update.

∙	OpenAPI spec (formerly Swagger) is a JSON/YAML file that auto-generates interactive documentation — tools like Mintlify render it beautifully.

∙	Idempotency keys let clients safely retry failed requests — important for payment and mutation endpoints.

∙	Webhook delivery (your API pushing to the client’s server) is the outbound counterpart — retry logic and signature verification matter here.

What you don’t need to know:

gRPC, GraphQL federation internals, or how to build a reverse proxy.

The fCTO signal: You can build the integration layer that makes a startup enterprise-ready. That’s a line item in a Series A technical roadmap.

Milestone: A live API with a docs page, working API key auth, and a rate-limited endpoint a developer could integrate in under 30 minutes.

PROJECT 13 — The Data Infrastructure Build

Why it matters for fCTO positioning: In NYC, especially in finance and media, data is the product. Investors at Series A will ask “what does your data infrastructure look like?” Most founding teams have no answer. You will.

What you’re building:

An end-to-end data pipeline for a real use case — ingest data from external sources (APIs, CSVs, webhooks), transform and clean it, store it in a structured warehouse, and surface it through an internal dashboard with AI-generated insights. Could be financial data, social media analytics, real estate data, market signals — pick a vertical relevant to NYC.

What this unlocks:

You understand the operational backend of a data business. Not just the user-facing app, but the infrastructure that feeds it. This is what separates a fCTO from a web developer in the eyes of a VC.

What you need to know (architect level):

∙	ETL (Extract, Transform, Load): extracting from a source, cleaning/normalizing it, loading into your warehouse. Airbyte or custom scripts handle the extract layer.

∙	A data warehouse (Supabase, BigQuery, or ClickHouse) is optimized for analytical queries — different from your transactional database.

∙	Cron jobs / scheduled tasks run your pipeline on a schedule — Vercel Cron or a simple server-side scheduler.

∙	Data quality checks: row counts, null checks, range validation — run these after every load and alert on failures.

∙	dbt (data build tool) transforms raw data into clean, documented tables — it’s SQL with version control and testing built in.

What you don’t need to know:

Spark, Hadoop, Kafka internals, or how to manage a data engineering team.

The fCTO signal: You can walk into a data-heavy startup and architect their entire data stack in the first 30 days. That is a specific, highly paid engagement type in NYC.

Milestone: A live pipeline pulling real external data, transforming it, and surfacing AI-interpreted insights on a dashboard. Write a one-page technical architecture doc explaining the system.

PROJECT 14 — The Technical Due Diligence Report

Why it matters for fCTO positioning: VCs, family offices, and acquirers in NYC pay fCTOs specifically for technical due diligence. This is a direct revenue service — $5K–$25K per engagement. Most people who call themselves fCTOs have never actually produced one.

What you’re building:

Pick a real open-source project or a startup with a public GitHub (many post on Show HN). Do a full technical audit and produce a formal report. Evaluate: code quality and architecture, scalability risks, security posture, technical debt, team velocity signals, infrastructure maturity, and a recommendation (invest / pass / invest with conditions).

What this unlocks:

You have a tangible, professional artifact that demonstrates you can evaluate a codebase like an expert — not just build one. This is a completely different capability signal and it’s directly monetizable.

What you need to know (architect level):

∙	Code quality signals: test coverage, linting config, PR size and frequency, commit message patterns — these are leading indicators of team discipline.

∙	Architecture red flags: no separation of concerns, business logic in the frontend, no environment separation (dev/staging/prod), secrets in code.

∙	Security basics: dependency vulnerability scanning (npm audit, Snyk), authentication patterns, data encryption at rest and in transit, input validation.

∙	Scalability questions: where are the bottlenecks? What breaks at 10x traffic? Is the database indexed properly?

∙	The report format: Executive Summary → Technical Findings (rated High/Medium/Low risk) → Recommendations → Appendix. 5–10 pages. Investor-grade writing.

What you don’t need to know:

How to run formal penetration tests, how to do SAST/DAST scanning, or how to perform cryptographic audits.

The fCTO signal: You can produce a document a VC hands to their investment committee. That is a service worth paying for and a portfolio piece that proves you operate at a different level than a contractor.

Milestone: A full technical due diligence report on a real codebase. PDF format, executive summary on page one. This is a portfolio document you send to every potential client.

PROJECT 15 — The Raise-Ready Technical Package

Why it matters for fCTO positioning: When a founder hires you as fractional CTO before a raise, the output they need is this: a complete technical package that makes investors confident the technology is real, scalable, and de-risked. Being able to produce this on demand is a premium service.

What you’re building:

For one of your previous projects (or a client project), produce the complete technical documentation package that a startup would present to investors and a technical advisor board: system architecture diagram, infrastructure diagram, security posture doc, technical roadmap (6-month), stack decision rationale document, and a one-page “why we won’t need a rewrite at scale” narrative.

What this unlocks:

You can walk into a pre-seed or seed-stage company and say “in 30 days I will have your entire technical story investor-ready.” That is a $10K–$30K engagement and founders will pay it because a bad technical story kills raises.

What you need to know (architect level):

∙	Architecture diagrams: use Excalidraw or Mermaid — show how data flows from user to frontend to API to database to third-party services.

∙	Infrastructure diagrams: show your cloud provider layout — which services live where, how they communicate, where your failure points are.

∙	Security posture doc: authentication method, data encryption, access controls, compliance considerations (SOC2 readiness, GDPR, HIPAA if relevant).

∙	Technical roadmap format: Now (0–30 days) / Next (30–90 days) / Later (90–180 days) — tied to business milestones, not just features.

∙	Stack decision rationale: for each major technology choice, explain why you chose it over the alternatives. Investors ask this. Founders often can’t answer it.

What you don’t need to know:

How to file for SOC2 certification, how to negotiate cloud contracts, or how to build a formal engineering org chart.

The fCTO signal: You speak the language of technical investors. You don’t just build — you de-risk. That is what fCTOs are paid for in NYC.

Milestone: A complete technical package PDF, designed cleanly, that you could drop into any investor data room today. This is your highest-value portfolio artifact.

The compressed modern stack summary

Here’s the summary version you actually need:

1. Static frontend / marketing / content-heavy sites

Use a static-first approach. These are fast, easy to deploy, and mostly about layout, copy, responsiveness, SEO, and light interactivity. Great when the site is mostly content and conversion.

2. Product web apps / dashboards / SaaS

Use Next.js App Router. Think of it as the default modern React full-stack shell for authenticated apps, dashboards, forms, admin areas, and mixed server/client rendering. Next explicitly positions App Router as the newer router supporting newer React features like Server Components.

3. Fast backend plumbing

Use Supabase when you want Postgres, auth, storage, realtime, APIs, and edge functions without stitching five vendors together. That is exactly the kind of leverage a one-man AI-native shop should exploit.

4. Edge/API/realtime coordination apps

Use Cloudflare Workers when you want low-friction APIs, edge execution, queues, KV, D1, or Durable Objects. Use Durable Objects when many users need shared state or coordination, like chat, collaborative editing, live rooms, or multiplayer-ish state.

5. AI-native workflow

Your advantage is not typing code faster. It is:

scoping the product correctly

choosing the right architecture

creating a clean project spec

AI-Native Builder Roadmap: Zero → Architect in 12 Weeks

Your profile: Former engineer turned AI-native architect. One-man shop. Idea → design → prototype → launch in weeks. You need to know what technologies can do, not how they work internally. Claude Code and Cursor execute. You direct.

PROJECT 1 — The Living Landing Page

Phase: Deploy | Timeframe: Day 1–2

Stack: Vercel, GitHub, Tailwind CSS, Framer Motion

What you’re building:

Take your HTML site. Add motion — scroll-triggered reveals, parallax, a hero with a looping gradient mesh. Deploy to Vercel in one command. Wire up a custom domain.

What this unlocks:

You now understand the deployment pipeline. Git push → build → CDN → URL. That’s it. Every project after this ships the same way.

What you need to know (architect level):

∙	Vercel is a CDN that auto-builds on push — you never touch a server.

∙	Tailwind is a utility class system — describe styling in HTML, not CSS files.

∙	Framer Motion is a React animation library — think CSS transitions but programmable.

∙	A custom domain is ~$12/yr and makes everything feel real.

What you don’t need to know:

Webpack config, DNS TTL values, or how Tailwind’s JIT compiler works.

Milestone: A live URL with your name on it that loads in under 1 second globally.

PROJECT 2 — The Dynamic App

Phase: Data | Timeframe: Week 1

Stack: Next.js, Supabase, Tailwind

What you’re building:

Build a simple tool — a public directory, a link aggregator, a reading list tracker. Anything where data is stored and retrieved. Users submit → data saves → page shows it.

What this unlocks:

You understand the request-response loop. Frontend asks, backend answers, database holds. The full loop. Every app is just variations of this.

What you need to know (architect level):

∙	Next.js is a React framework — it handles routing, SSR, and API routes in one project.

∙	Supabase is a hosted Postgres database with a REST API auto-generated from your schema — no backend code needed for basic CRUD.

∙	Row-level security means you can define who sees what data, directly in the database.

∙	The difference between client-side and server-side rendering matters for SEO and load speed.

What you don’t need to know:

SQL joins, database normalization theory, or how Next.js compiles React.

Milestone: A live app where strangers can submit data and see it on screen.

PROJECT 3 — The Members Club

Phase: Auth | Timeframe: Week 2

Stack: Clerk, Next.js, Supabase

What you’re building:

Add auth to your previous project. Users sign up, log in, have a profile page. Protect certain routes. Show user-specific data. Add a simple role: admin vs user.

What this unlocks:

You understand identity. Every SaaS is built on this. Once you can gate access, you can charge for it.

What you need to know (architect level):

∙	Clerk is auth-as-a-service — signup, login, MFA, social OAuth, all handled for you.

∙	JWT tokens are how the frontend proves identity to the backend — you don’t build these.

∙	Protected routes mean certain pages redirect to login if you’re not authenticated.

∙	User metadata lets you store things like plan tier, onboarding status, preferences.

What you don’t need to know:

How bcrypt works, how OAuth flows are structured, or how JWTs are signed.

Milestone: A gated app where only logged-in users see certain content.

PROJECT 4 — The AI-Powered Tool

Phase: AI | Timeframe: Week 2–3

Stack: Claude API / OpenAI API, Next.js, Vercel AI SDK

What you’re building:

Build something where AI is the core feature — a document analyzer, a writing assistant, a cold email generator, a resume reviewer. User inputs something, AI processes it, output is valuable enough to screenshot.

What this unlocks:

You understand AI as an API call. Prompt in, completion out. Streaming makes it feel alive. This is the building block of every AI product.

What you need to know (architect level):

∙	The Vercel AI SDK handles streaming responses so text types out in real time — critical for UX.

∙	System prompts define the AI’s persona and constraints — this is your core product logic.

∙	Temperature controls creativity vs. consistency — low for factual, high for creative.

∙	Token limits matter: ~750 words per 1000 tokens. Design prompts to be efficient.

∙	Structured outputs let you tell the AI to respond in JSON — so you can render dynamic UI from its response.

What you don’t need to know:

Transformer architecture, fine-tuning, embeddings math, or model weights.

∙	Optimistic UI means updating the screen immediately before the server confirms — feels instant.

WebSocket protocol details or how to build your own pub/sub system.

Zero to mvp

The AI-Native SDLC

From Idea on Paper to MVP Live — The Complete Protocol

Who this is for: A solo architect/fCTO operating with Claude Code, Cursor, and the modern AI stack. Every phase has a defined input, a defined output, and the exact AI tooling to execute it.

PHASE 0 — IDEA CAPTURE

The Raw Idea Doc

Input: A thought, a problem, a gap you noticed.

Output: A single-page Idea Brief.

Time: 15–30 minutes.

Tool: Claude (chat), Notion, or a plain text file.

The prompt to run on your raw idea:

“I have an idea: [describe it in 3 sentences]. Play devil’s advocate. What’s the core problem being solved, who has it, what does a solution actually need to do at minimum, and what’s the single biggest reason this fails? Then give me a one-paragraph version that a non-technical founder could read and say ‘yes, build that.’”

What your Idea Brief must contain:

∙	Problem statement (one sentence, no jargon)

∙	Who has this problem and how often (specificity matters — “Series A SaaS founders in NYC” beats “businesses”)

∙	What the user does today without your product (the workaround)

∙	The core value exchange (what they get, what they give)

∙	The riskiest assumption (the one thing that, if wrong, kills the idea)

∙	Out of scope (what this is NOT, even if it seems related)

Gate check: If you can’t write the problem statement in one sentence, you don’t understand the idea yet. Do not proceed.

PHASE 1 — PROJECT BRIEF

Translate the Idea into a Buildable Scope

Input: Your Idea Brief.

Output: A Project Brief document (2–4 pages).

Time: 1–2 hours.

Tool: Claude (chat) + your own judgment.

The prompt:

“Here is my Idea Brief: [paste it]. Turn this into a full Project Brief. Include: product vision, target user persona, core user journey (the happy path only), MVP feature list ranked by necessity, explicitly excluded features, success metrics for MVP launch, and a risk register with the top 3 technical and 3 product risks.”

Project Brief structure:

1. Product Vision

One paragraph. What does this become if it works? Not the MVP — the thing the MVP is the first step toward.

2. Target User Persona

Name them. Give them a job, a frustration, a goal. “Marcus, 34, technical co-founder at a pre-seed fintech startup. He’s trying to ship before his runway runs out and his co-founder keeps changing the product spec.” Real people, not demographics.

3. Core User Journey (Happy Path)

Write this as a numbered flow. User arrives → user does X → system does Y → user achieves Z. No edge cases. No error states. Just the path that works perfectly.

4. MVP Feature List

Three tiers:

∙	Must Have (MVP dies without these)

∙	Should Have (MVP is weak without these)

∙	Won’t Have (explicitly banned from this sprint)

5. Success Metrics

Define what “launched and working” looks like in numbers. Not vanity metrics. Examples: 50 users complete the core action, <3 second load time, 0 data loss incidents, 10 paying customers in week one.

6. Risk Register

|Risk                   |Likelihood|Impact|Mitigation                         |

|-----------------------|----------|------|-----------------------------------|

|Auth provider outage   |Low       |High  |Fallback email login               |

|AI API latency too high|Medium    |High  |Stream responses, add loading state|

Gate check: Show the Project Brief to one person who is your target user. If they say “yes, I would use that” — proceed. If they say “hm, but what about…” — update the brief first.

PHASE 2 — TECHNICAL SPECIFICATION

Translate the Brief into Engineering Language

Input: Approved Project Brief.

Output: Technical Spec document (5–10 pages).

Time: 2–4 hours.

Tool: Claude (chat) — this is the most important prompt in the whole pipeline.

The master prompt:

“You are a senior software architect. Here is my Project Brief: [paste it]. Produce a complete technical specification. Include: recommended tech stack with justification for each choice, full data model (all entities, fields, relationships, and types), API route design (every endpoint, method, inputs, outputs), authentication and authorization flow, third-party integrations needed, environment architecture (local/staging/production), file and folder structure, non-functional requirements (performance targets, security requirements, scalability considerations), and open technical questions that need answers before building begins. Be opinionated. Make decisions. Flag where I have choices and what the tradeoff is.”

Technical Spec structure:

1. Stack Decision

For each layer, state the chosen technology and why. Format:

∙	Frontend: [technology] — [why this over alternatives]

∙	Backend/API: [technology] — [why this over alternatives]

∙	Database: [technology] — [why this over alternatives]

∙	Auth: [technology] — [why this over alternatives]

∙	Hosting/Infra: [technology] — [why this over alternatives]

∙	AI layer: [technology/model] — [why this over alternatives]

∙	Email: [technology] — [why this over alternatives]

∙	Payments (if applicable): [technology]

∙	Analytics: [technology]

2. Data Model

Every entity as a table. Every field with its type, constraints, and relationships. Example format:

users

id: uuid (PK)

email: string (unique, not null)

created_at: timestamp

plan_tier: enum('free', 'pro', 'enterprise')

org_id: uuid (FK → organizations.id)

organizations

id: uuid (PK)

name: string

created_at: timestamp

stripe_customer_id: string

Draw relationships explicitly: “users belongs_to organizations, organizations has_many users.”

3. API Design

Every route. Format:

POST /api/auth/signup

Input: { email, password, org_name }

Output: { user_id, token, org_id }

Auth: public

Side effects: creates user, creates org, sends welcome email

GET /api/documents/:id

Input: document id in path

Output: { id, title, content, created_at, owner }

Auth: authenticated, must be member of owning org

Side effects: none

4. Auth Flow

Draw this as a sequence. Who calls what in what order. Where does the token come from, where does it go, how is it validated on protected routes.

5. Third-Party Integrations

For each: what triggers it, what data flows, what happens if it fails, what the fallback is.

6. Environment Architecture

∙	Local: what runs locally, what points to dev cloud services

∙	Staging: mirrors production, used for QA before every deploy

∙	Production: the live thing

Define which environment variables exist and what they control.

7. Folder Structure

Write out the full directory tree before a single line of code is written. This forces architecture decisions upfront.

/app

/api          → API routes

/components   → Reusable UI components

/lib          → Utility functions, API clients, helpers

/hooks        → Custom React hooks

/types        → TypeScript type definitions

/supabase

/migrations   → Database schema migrations

/seed         → Seed data for development

/docs           → Architecture docs, ADRs

8. Open Questions

A numbered list of decisions not yet made. Assign each one an owner and a deadline. Nothing proceeds past Phase 3 until every open question has an answer.

Gate check: The technical spec is complete when a developer who has never heard of this project could read it and start building without asking you a single clarifying question. If that’s not true, it’s not done.

PHASE 3 — ARCHITECTURE & SYSTEM DESIGN

The Blueprint Before the Build

Input: Approved Technical Spec.

Output: Architecture diagrams + Architecture Decision Records (ADRs).

Time: 2–3 hours.

Tool: Claude (chat for content), Excalidraw or Mermaid for diagrams.

What to produce:

1. System Architecture Diagram

Shows how the major components of your system connect. Every box is a service or layer. Every arrow is data flow with a label. Must include: user, frontend, API layer, database, auth provider, AI provider, any third-party services. Tools: Excalidraw (visual) or Mermaid (code-generated, version-controllable).

Prompt to generate a Mermaid diagram:

“Based on this technical spec, generate a Mermaid system architecture diagram showing all components and data flow: [paste spec]”

2. Data Flow Diagrams

For the 2–3 most complex user journeys (signup, core action, payment). Show exactly what happens at each step — which service is called, what data is passed, what is stored.

3. Infrastructure Diagram

Where does everything live? Which Vercel project, which Supabase project, which external services. Show the network topology at a high level. This matters for security (what’s public-facing) and for cost (what’s always running vs. serverless).

4. Architecture Decision Records (ADRs)

For every significant technical decision, write a one-paragraph ADR. Format:

ADR-001: Using Supabase over PlanetScale

Status: Decided

Context: We need a hosted Postgres database with row-level security

for multi-tenancy and real-time capabilities for live features.

Decision: Supabase. It provides RLS natively, a Realtime subscription

layer, and auto-generated REST/GraphQL APIs.

Consequences: We are coupled to Supabase's pricing and uptime.

Migration cost later would be high.

Alternatives considered: PlanetScale (no RLS), Neon (no realtime),

self-hosted Postgres (ops overhead).

ADRs live in your /docs folder and become the technical memory of the project. Invaluable for due diligence and onboarding.

Gate check: Could a new technical hire understand your entire system in 30 minutes using only these diagrams and ADRs? If yes, proceed.

PHASE 4 — DEVELOPMENT ENVIRONMENT SETUP

Before the First Line of Code

Input: Approved architecture.

Output: A running local environment, initialized repo, and scaffolded project structure.

Time: 1–3 hours.

Tool: Claude Code (primary), GitHub.

The setup protocol:

1. Repository Initialization

- Create GitHub repo (private)

- Initialize with README, .gitignore, LICENSE

- Set up branch protection on main: require PR review, no direct pushes

- Create branches: main (production), staging, dev (working branch)

2. Project Scaffolding

Use Claude Code or the framework’s CLI to scaffold the project structure exactly as defined in your Technical Spec folder structure. Do not improvise here — follow the spec.

Claude Code prompt for scaffolding:

“Initialize a Next.js 14 project with TypeScript, Tailwind CSS, and the App Router. Create the following folder structure: [paste from spec]. Add the following dependencies: [list from spec]. Create placeholder files for each major component with a comment describing what it will do. Do not write any feature logic yet.”

3. Environment Variables

Create .env.local with all variables from your spec — even if the values are placeholder. Create .env.example (no real values) and commit it. Never commit .env.local.

4. Database Initialization

Run your first Supabase migration to create the schema from your data model. This is the ground truth of your database — every change after this is a migration file, never a manual edit.

Claude Code prompt:

“Generate Supabase migration SQL files for this data model: [paste data model from spec]. Create separate migration files for: initial schema, row-level security policies, indexes, and seed data.”

5. Linting and Formatting Setup

ESLint + Prettier configured before any code is written. Non-negotiable. Add a pre-commit hook (Husky) that runs lint and type-check before every commit. This prevents a category of bugs entirely.

6. CI/CD Pipeline

Connect GitHub to Vercel. Every push to dev gets a preview URL. Every merge to staging deploys to your staging environment. Every merge to main deploys to production. Set this up before writing any features.

Gate check: npm run dev starts the project with no errors. The database schema exists and is correct. All environment variables are documented.

PHASE 5 — FEATURE DEVELOPMENT

The Build Loop

Input: Scaffolded project + Technical Spec feature list.

Output: Working features, committed to dev branch.

Time: Variable per feature.

Tool: Claude Code for generation, Cursor for editing and iteration.

The feature development protocol:

1. Feature Branch

Every feature gets its own branch off dev.

git checkout dev

git pull

git checkout -b feature/user-authentication

Never build two features on the same branch.

2. The Claude Code Feature Prompt (the most important skill)

Structure every feature prompt the same way:

Context: [Brief description of the project and stack]

What exists: [What's already built that this feature connects to]

What I need: [Specific feature description]

Constraints:

- Follow the existing file structure (paste folder tree)

- Use TypeScript strict mode

- Follow this data model (paste relevant tables)

- These API routes need to be created (paste from spec)

- Use existing components from /components where possible

Output format: Create the following files (list them)

Do not: Create new dependencies without asking first

The specificity of this prompt is the difference between working code and spaghetti. Claude Code is as good as the context you give it.

3. The Review Before Accepting

Before accepting any generated code, check:

∙	Does it follow the folder structure from the spec?

∙	Are TypeScript types correctly defined (no any)?

∙	Are there hardcoded values that should be environment variables?

∙	Does it handle loading states, error states, and empty states?

∙	Does it follow the RLS policies defined in the spec?

4. Cursor for Iteration

Use Cursor for: debugging specific errors, refactoring a function, explaining what generated code does, making targeted edits to existing files. Claude Code generates, Cursor refines.

Cursor prompt patterns:

∙	“Explain what this function does and identify any edge cases it doesn’t handle”

∙	“Refactor this to follow the repository pattern”

∙	“This is throwing [error]. Here is the relevant code and the full error message. Fix it.”

∙	“Add TypeScript types to this file without changing any logic”

5. Commit Protocol

Commit frequently. Every logical unit of work is a commit. Commit message format:

feat: add user authentication with Clerk

fix: resolve infinite loop in useEffect on dashboard

chore: add environment variable for Resend API key

docs: update README with local setup instructions

This is conventional commits format. It makes your git history readable and enables automated changelog generation.

Feature completion checklist (before PR):

∙	Feature works end-to-end in the happy path

∙	Loading state implemented

∙	Error state implemented

∙	Empty state implemented

∙	Mobile responsive (if applicable)

∙	No console errors or warnings

∙	TypeScript compiles with no errors

∙	No hardcoded strings that should be env variables

∙	RLS policies tested (try accessing data as the wrong user)

PHASE 6 — TESTING PROTOCOL

The Minimum Viable QA Stack

Input: Completed feature branch.

Output: Tested, reviewed code ready to merge.

Tool: Claude Code for test generation, Playwright for E2E.

The three-layer testing approach for solo AI-native builders:

Layer 1: Type Safety (always on)

TypeScript is your first test layer. If it compiles cleanly with strict mode, a class of bugs is already eliminated. Run tsc --noEmit as part of your pre-commit hook. This is non-negotiable.

Layer 2: Unit Tests for Critical Logic

You do not need to unit test everything. You need to unit test: anything that handles money, anything that handles auth and permissions, any complex data transformation, any function with multiple code paths.

Claude Code prompt for test generation:

“Write Jest unit tests for this function: [paste function]. Test: the happy path, edge cases including null/undefined inputs, error conditions, and boundary values. Use descriptive test names that explain what is being tested.”

Layer 3: End-to-End Tests for Critical Flows

Use Playwright. Write E2E tests for: user signup and login, the core feature (the thing your product does), payment flow, anything that, if broken, would cause user loss or revenue loss.

Claude Code prompt:

“Write a Playwright E2E test for this user flow: [describe the flow]. The test should: navigate to the page, complete each step, assert the expected outcome at each stage, and verify the final state in the database by checking the UI response.”

The testing minimum for MVP launch:

∙	TypeScript strict mode passing: yes/no

∙	Core user journey E2E test: passing

∙	Auth flow E2E test: passing

∙	Payment flow test (if applicable): passing

∙	Manual smoke test on staging: completed

What you are NOT doing at MVP stage:

100% test coverage, load testing, chaos engineering, full accessibility audit. Those are post-launch. Ship first.

PHASE 7 — CODE REVIEW PROTOCOL

The AI-Augmented Review

Input: Completed feature branch.

Output: Reviewed and approved code.

Tool: Claude (chat) for AI review, GitHub for PR.

The AI code review prompt:

Before opening a PR, paste your diff or key files into Claude and run:

“Review this code as a senior engineer. Check for: security vulnerabilities (especially around auth and data access), performance issues, potential race conditions, missing error handling, TypeScript type safety violations, deviation from standard patterns, and anything that will break under load or edge cases. Be specific. Line numbers if possible.”

What to look for in the AI review output:

∙	Any security flag is a must-fix before merge

∙	Performance issues are fix-before-merge if they affect core flows

∙	Style and pattern issues are fix-before-merge to maintain consistency

∙	Everything else is logged as a future ticket

PR checklist:

∙	Branch is up to date with dev (no merge conflicts)

∙	AI code review completed, critical issues resolved

∙	Feature checklist from Phase 5 completed

∙	Tests passing locally

∙	Preview deployment URL attached to PR (Vercel auto-generates this)

∙	PR description explains what was built and why, not just what changed

Merge protocol:

Squash merge into dev. This keeps the git history of dev clean — one commit per feature, not 47 “fix typo” commits.

PHASE 8 — STAGING VALIDATION

Before Any Merge to Main

Input: Feature merged to dev, promoted to staging.

Output: Signed-off staging build ready for production.

Time: 30–60 minutes per major feature set.

Tool: Manual testing + Playwright E2E suite against staging URL.

The staging protocol:

1. Deploy to staging

Merge dev into staging. Vercel auto-deploys. Staging uses real cloud services but separate instances — a separate Supabase project, Stripe test mode, Clerk test instance.

2. Smoke test checklist (run on every staging deploy)

∙	Sign up as a new user works end to end

∙	Log in / log out works

∙	Core feature works (the thing the product does)

∙	Navigation and routing correct

∙	No broken images, fonts, or assets

∙	Mobile view is correct (use browser dev tools device emulator)

∙	Error states display correctly (try to break it deliberately)

∙	Payment flow completes in Stripe test mode (if applicable)

∙	Emails are received (sign up, welcome, etc.)

∙	Data is being stored correctly (check Supabase table editor)

3. Performance check

Run Lighthouse on the staging URL. Target scores: Performance > 85, Accessibility > 90, Best Practices > 90, SEO > 90. Anything under 85 on Performance needs to be investigated before launch.

4. Security check

∙	Try to access routes you shouldn’t be able to access (while logged out, while logged in as wrong user)

∙	Check network tab: are you leaking sensitive data in API responses?

∙	Check for exposed environment variables in client-side code (they’d appear in browser source)

Gate check: Every item on the smoke test checklist passes. Performance score is acceptable. No security holes found. This is the only gate to production.

PHASE 9 — PRODUCTION DEPLOYMENT

The Go-Live Protocol

Input: Approved staging build.

Output: Live production deployment.

Time: 15–30 minutes.

Tool: Vercel, GitHub.

Pre-deploy checklist:

∙	All environment variables set in Vercel production environment

∙	Production Supabase project has correct schema (run migrations)

∙	Stripe is in live mode (not test mode) in production env

∙	Custom domain configured and SSL active

∙	Error monitoring active (Sentry or equivalent — set up before launch, not after)

∙	Analytics active (PostHog — set up before launch so day one data is captured)

∙	Staging smoke test passed within the last 24 hours

The deploy:

git checkout main

git merge staging

git push origin main

Vercel detects the push and builds. Watch the build log in real time. Build failure = do not proceed. Investigate and fix before retrying.

Post-deploy validation (do this within 10 minutes of deploy):

∙	Production URL loads correctly

∙	Sign up flow works on production

∙	Core feature works on production

∙	Payments work in live mode (run a real $1 test transaction)

∙	Check error monitoring dashboard — zero new errors

∙	Check analytics — events are being tracked

If something is broken in production:

Revert immediately. Do not try to hotfix in production.

git revert HEAD

git push origin main

Fix on dev, re-test on staging, re-deploy. Never fix production directly.

PHASE 10 — LAUNCH PROTOCOL

Getting to Your First 100 Users

Input: Live, validated production build.

Output: Real users using the product.

Tool: Product Hunt, social, direct outreach, email.

The 72-hour launch sequence:

Day before launch:

∙	Draft your Product Hunt post (tagline, description, first comment explaining your story)

∙	Create 3 screenshots (the before state, the product, the outcome/result)

∙	Create a 60-second demo GIF or video (Loom works)

∙	Set up your first email sequence in Resend — welcome email triggers on signup

∙	Seed your database with realistic demo data so new users aren’t staring at empty states

∙	Add an “early access” or founding member offer — first 50 users get X free

∙	Message 20 people in your network who are target users. Not “please upvote.” “I built something for people with your problem. Can I send it?”

Launch day (Product Hunt posts at 12:01am PST):

∙	Post goes live

∙	Share to every relevant community simultaneously: relevant Slack groups, Twitter/X, LinkedIn, relevant subreddits

∙	Your 20 network contacts get a direct message: “it’s live, here’s the link”

∙	Reply to every single comment on Product Hunt within the first 2 hours

∙	Watch your error monitoring and analytics obsessively for the first 6 hours

Post-launch (48 hours after):

∙	Review every user session in PostHog — where did people drop off?

∙	Check your Supabase tables — what actions are users actually completing?

∙	Message every person who signed up but didn’t complete the core action — ask them why

∙	Write down every piece of feedback you received — do not act on any of it yet

∙	Identify the one thing that, if fixed, would have the highest impact on activation

∙	Open a new feature branch and start the loop again

THE COMPLETE SDLC SUMMARY

Phase 0:  Idea Capture          → Idea Brief (30 min)

Phase 1:  Project Brief         → Scoped product document (2hr)

Phase 2:  Technical Spec        → Engineering blueprint (4hr)

Phase 3:  Architecture          → Diagrams + ADRs (3hr)

Phase 4:  Dev Environment       → Repo + scaffold + DB (3hr)

Phase 5:  Feature Development   → Working code on dev branch (variable)

Phase 6:  Testing               → Types + unit + E2E (1hr per feature)

Phase 7:  Code Review           → AI review + PR approval (1hr per feature)

Phase 8:  Staging Validation    → Smoke test + performance (1hr per deploy)

Phase 9:  Production Deploy     → Live build + post-deploy validation (30min)

Phase 10: Launch                → Users, feedback, iteration loop (72hr)

Total time from idea to MVP launch: 2–6 weeks depending on product complexity. For a focused AI-native builder using Claude Code and Cursor: consistently closer to 2.

The Governing Principle

Every phase has a gate. The gate is a binary question: is the output of this phase complete enough that the next phase will not fail because of it? If yes, proceed. If no, finish the phase. The reason most MVPs fail or take too long is not that the builders were too slow in the build phase. It is that they skipped the spec, rushed the architecture, and then spent 80% of their time debugging decisions they never consciously made.

The AI tools collapse the execution time by 10x. The phases still exist. The thinking still has to happen. You are the architect. The documents you produce in Phases 0–3 are your highest-leverage work. Everything after that is execution that Claude Code can largely handle.​​​​​​​​​​​​​​​​
