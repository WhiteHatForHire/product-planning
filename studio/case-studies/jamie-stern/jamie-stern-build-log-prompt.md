# Jamie stern build log prompt

> Imported from the source archive and converted to Markdown for searchability. Treat the source path as provenance, not as the current canonical location.
Generate a Field Log entry for the Jamie Stern engagement in the voice

and structure of Marcus Vale's Builder's Log entries.

Context:

This is the private working log for a fractional CTO engagement on a

WordPress migration and stabilization project. The site is a luxury

fabric/hospitality brand. The work is being done alongside Sam (point

of contact) and an Indian dev team running the WP Engine staging build.

This log is for me and for the eventual client-engagement case study,

not for the client.

Format and tone:

- First person, reflective but grounded. Not journaling-as-therapy,

not hype, not status report.

- Mix of narrative ("started this morning at...") with bulleted facts

where they earn it, and direct callouts of discipline moments.

- No em dashes anywhere in the entry.

- No horizontal divider lines.

- Heading hierarchy: bold section headers, no markdown # signs, plain

bold sentence-style headers.

- Voice: a fractional CTO who is also a writer. Honest about fatigue,

scope creep, side-quest temptations, mistakes caught and recovered.

Names where the discipline held and where it almost didn't.

- Avoid "just vibes," Gen Z "vibes," and AI-tell phrases.

Required sections (in this order, omit any that don't apply to the day):

1. Opening orientation. What day of the engagement, what phase

(audit / stabilization / proof-strike / handoff), what the original

plan for the session was.

2. What actually happened, more or less in order. Narrative, not

bullets. Include the messy parts: tools breaking, agent

misbehavior, false starts, side-quest temptations, mistakes caught

and recovered. Specifically note moments where the agent overstepped

(squashing history, fabricating output, etc.) and how I responded.

3. Specific named moments worth banking. Pull out 2-4 things that

would be easy to forget but matter to the case study or to

future-me. Each one gets a short paragraph, not a bullet. These

are the moments where the Director Model played out in practice,

for better or worse.

4. What I'm noticing about the work. Reflective, pattern-level.

Discipline-over-speed framing where it applies. The fCTO

positioning specifically. Greek vocabulary if it earns its place

(Phronesis, Sophrosyne, Aidōs, Kairos, Techne) — never forced.

Aristotelian framing where it fits.

5. Where the engagement is now. Precise about phase progress and

what "done" actually means at this stage. Distinguish "code

shipped" from "client value delivered." Distinguish "I know what

to do" from "it has been done."

6. A few things to lock in before stopping. Decision debt. Things

deliberately parked. What's pinned for later vs. what's done. Each

parked thing named explicitly. Include any open questions for Sam

that I am not asking yet.

7. Cost ledger update if costs were incurred this session. Token

spend, Replit credits, GitHub usage, time. Running tally for

case study purposes.

8. What I'm taking with me on the break. What I want to be true at

end of session. The win framing. Specifically, what proof-strike

evidence I now have to send Sam.

9. Optional: Note to future Marcus. Direct address. What you want

future-you reading this later to remember about today's

engagement. One paragraph max.

Inputs to use:

- The day's actual events (commits, PRs, deployments, fixes)

- The discipline calls (where I almost spun out and didn't, or where

I did spin out and recovered)

- The agent governance calls (when an agent overstepped and how I

redirected)

- Cost ledger updates (Claude Code tokens, cloud session usage, API

spend, time)

- Any commits, PRs, or deployment milestones with their exact

identifiers (commit hashes, PR numbers, branch names)

- Side-quest temptations resisted (this is the case study evidence,

name them explicitly)

- Client communication moments (drafted, sent, deliberately not yet

sent)

What NOT to do:

- Don't manufacture drama. If a day was clean and uneventful, the

entry is short and clean.

- Don't compress timeline. If it took 90 minutes, say 90 minutes.

- Don't claim more than was done. If a PR shipped but isn't merged,

say "shipped, not merged." If a doc was drafted but not sent, say so.

- Don't write a case study in disguise. This is the private log. The

public case study is downstream.

- Don't include sensitive client information that shouldn't appear in

even a private log (real production credentials, full API keys,

client internal tensions). Reference them obliquely if needed.

- Don't add em dashes, divider lines, or generic AI-style closing

flourishes.

Stop after the entry. No meta-commentary about what was generated.
