Marcus's Serious Build Checklist

A practical operating system for AI-native rapid deployment

Draft 1 | April 25, 2026

Move fast, but every serious build gets a boring checklist.

Purpose

This document is a guardrail for rapid AI-assisted building. It protects speed from becoming chaos. It gives every real build a clear user, a bounded V1, a budget cap, safety limits, test coverage, and an operational owner.

Use it before you build apps, automations, AI workflows, prototypes, client tools, internal dashboards, wellness systems, or anything that touches real users, money, private data, health-adjacent claims, or public reputation.

1. The Operating Principle

AI changes the unit of learning. The old path was courses, syntax memorization, and years of framework immersion. The new path is shipped systems, tight prompts, verification loops, and reusable architecture. But AI does not remove responsibility. It shifts the human role from typing every line to directing the system with taste, judgment, sequencing, and verification.

The danger is not ambition. The danger is unbounded speed. The answer is not to slow down until the fire dies. The answer is to keep the fire inside a forge.

Core identity

Marcus is not training to become a syntax monkey. Marcus is training to become an AI-native systems orchestrator with enough engineering experience to know what must be verified.

The Greek lens

Techne: craft, tools, implementation, and the ability to make the thing real.

Phronesis: practical judgment about what to build, when to build it, and why it matters.

Praxis: learning through repeated action, not passive consumption.

Sophrosyne: right measure, boundaries, budgets, and restraint.

Arete: excellence through clean reps, feedback, and refinement.

Hubris: speed without verification, scope without limits, and confidence without evidence.

2. What Counts as a Serious Build

A serious build is any project where failure matters beyond private learning. The more real the user, the more boring the checklist must become.

[ ] A real person will use it.

[ ] A client, partner, or local operator will depend on it.

[ ] It may create or influence revenue.

[ ] It stores private, personal, business, health-adjacent, recovery, relationship, location, payment, or identity data.

[ ] It gives advice, matches people, routes opportunities, drafts messages, or makes recommendations.

[ ] It affects Marcus's public reputation as a builder, founder, or fractional CTO.

The three build modes

Practice Build: A learning rep. Real money and private data should not depend on it. Use small budgets, fast experiments, and aggressive deletion of unnecessary features.

Pilot Build: A controlled test with real users. Human supervision is mandatory. Reliability, privacy, onboarding, support, and feedback collection matter.

Production or Client Build: A serious system with real operational consequences. It needs documentation, support ownership, rollback plans, security basics, and explicit client/user expectations.

3. The Ten Gates

Do not treat these as bureaucracy. Treat them as load-bearing questions. If a gate cannot be answered, the build is not blocked forever, but the next move should become discovery, not implementation.

Gate 0: Mode and Motive

A good build starts with a truthful motive. Building from energy is fine. Building from urgency, avoidance, or a need to prove yourself should trigger a smaller scope, not a bigger one.

[ ] What mode am I in right now: Exploration, Integration, Build, or Arena?

[ ] Am I building from clarity, or am I building to escape discomfort?

[ ] Is this a conscious learning rep, a real pilot, or a production/client system?

[ ] What would a clean stopping point look like in 72 hours?

[ ] What would I regret not bounding before I start?

Gate 1: User, Problem, and Outcome

No specific user means no serious build. It can still be a practice build, but it is not yet a product.

[ ] Who is the exact user? Name a real person or real category.

[ ] What painful, repeated, expensive, confusing, or emotionally charged problem are they trying to solve?

[ ] What do they do today without this tool?

[ ] What would success look like after one use?

[ ] What would make them tell another person about it?

Gate 2: Smallest Valuable Loop

A serious V1 is not a tiny version of the dream. It is the smallest loop that proves the dream has a pulse.

[ ] What is the smallest useful version?

[ ] What is the input, transformation, output, and next human action?

[ ] Can the core loop work manually or semi-manually before automation?

[ ] What features are explicitly out of scope for V1?

[ ] What is the smallest visible win the user can feel?

Gate 3: Data and Trust

Data minimization is not just legal caution. It is product elegance. If the system does not need the data, do not collect it.

[ ] What data is collected?

[ ] What data is truly necessary?

[ ] What data should never be collected in V1?

[ ] Where is the data stored?

[ ] How can a user delete, correct, or avoid sharing sensitive information?

Gate 4: AI Role and Safety Boundary

The safest AI systems are honest about their role. They ask, explain, route, draft, and escalate. They do not pretend to be doctors, lawyers, therapists, prophets, or autonomous authorities.

[ ] Is the AI classifying, matching, drafting, summarizing, coaching, recommending, or routing?

[ ] Where can AI be wrong without serious harm?

[ ] Where must a human approve before action is taken?

[ ] What should the AI never claim?

[ ] What fallback appears when the AI is uncertain, unsafe, or outside scope?

Gate 5: Architecture and Implementation

Architecture should be boring where possible. Save creativity for the product insight, the user experience, and the system design.

[ ] What stack is being used, and why is it sufficient for V1?

[ ] What are the database tables and required fields?

[ ] What API keys, environment variables, and secrets are needed?

[ ] What happens when an API fails?

[ ] What logs, admin views, and error states are needed to debug the system?

Gate 6: Test Plan and Verification

Verification is the difference between directing the flow and being dragged by it.

[ ] What are the five happy paths?

[ ] What are the ten obvious edge cases?

[ ] What happens with empty, weird, hostile, duplicated, or incomplete inputs?

[ ] Does the main flow work on mobile?

[ ] What smoke test should run after every meaningful change?

Gate 7: Launch and Distribution

A beautiful empty app is not a product. Distribution is part of the build.

[ ] How will the first 10 users find this?

[ ] What exact message explains the offer in one sentence?

[ ] What onboarding step must be frictionless?

[ ] Who handles questions, complaints, refunds, or confusion?

[ ] What proof do we need before spending more credits?

Gate 8: Operations and Ownership

Boring ops are where serious builders separate from app tourists.

[ ] Who checks the system daily or weekly?

[ ] Who handles manual exceptions?

[ ] Where do issues get logged?

[ ] What needs a backup, export, or rollback path?

[ ] What recurring work does this create after launch?

Gate 9: Budget, Timebox, and Stop Condition

Credits are tuition, not casino chips. Spend to learn, validate, and ship. Do not spend to chase an undefined feeling of completion.

[ ] What is the Replit or AI credit cap?

[ ] What is the session timebox?

[ ] What are the allowed reasons to exceed the cap?

[ ] What result justifies another build pass?

[ ] What result means pause, kill, or simplify?

Gate 10: Post-Build Review

The build is not complete when the feature works. It is complete when the lesson has been harvested.

[ ] What shipped?

[ ] What actually worked?

[ ] What broke?

[ ] What did users do that surprised us?

[ ] What reusable component, prompt, schema, or pattern did this create?

4. Red Flags and Green Flags

Pause or shrink the build if these are true

[ ] The user is vague: tourists, businesses, everyone, creators, healers, founders.

[ ] The app needs a two-sided marketplace before the manual version has worked.

[ ] The AI is expected to act autonomously in a high-trust situation.

[ ] The system collects sensitive information without a clear reason.

[ ] The feature list keeps growing before the core loop is proven.

[ ] There is no path to the first 10 users.

[ ] There is no budget cap or timebox.

[ ] The phrase 'we can monetize later' is doing too much work.

[ ] The build is being used to avoid a hard conversation, a rest need, or a simpler business task.

Proceed if these are true

[ ] A manual or concierge version can work before full automation.

[ ] The first 10 users can be named, reached, or recruited directly.

[ ] The system creates an immediate benefit even if it never becomes a huge company.

[ ] The AI role is bounded and explainable.

[ ] The no-go list is clear.

[ ] The budget and stop condition are written down before the build begins.

[ ] The build creates reusable infrastructure for future products or client work.

[ ] Every human contributor has a defined role, compensation path, or learning agreement.

6. Example: Bali Wellness Connector

This example shows how the checklist applies to the wellness matching idea: massage therapists, healers, bodyworkers, breathwork facilitators, and other local providers serving tourists, expats, and residents in Bali.

Good V1 frame

- Concierge-backed matching system, not a giant marketplace.

- User fills out a simple intake: location, need, budget, timing, preference, contraindications they voluntarily disclose, and language preference.

- AI classifies the request and suggests provider categories.

- A human operator reviews the match, routes the request, and follows up.

- The system learns from completed bookings, feedback, cancellations, and provider reliability.

No-go scope for V1

[ ] No medical diagnosis or treatment claims.

[ ] No claim that a healer can cure trauma, addiction, illness, or psychiatric conditions.

[ ] No automatic provider approval without human vetting.

[ ] No fully autonomous booking decisions in sensitive or ambiguous cases.

[ ] No payment custody unless the legal, refund, and dispute workflow is clear.

[ ] No giant review platform before the first manual matches are proven.

Roles

Marcus: Product strategy, rapid build, AI matching logic, data model, admin workflow, safety boundaries, testing, and iteration.

Yuni: Provider recruiting, local trust, communication, quality signals, customer follow-up, and cultural interpretation.

Shared responsibility: Standards, pricing logic, provider rules, customer promise, and decisions about what is safe to claim.

Validation targets

[ ] Recruit 20 credible providers.

[ ] Create a simple provider profile database with categories, location, availability, price range, language, and quality notes.

[ ] Match 10 real customers or serious inquiries.

[ ] Complete at least 3 bookings or equivalent strong proofs of demand.

[ ] Collect feedback from both sides within 24 hours.

[ ] Decide after the pilot whether to continue, simplify, niche down, or kill.

7. The Serious Build Ritual

Before building

[ ] Write a one-page build brief.

[ ] Name the mode: practice, pilot, or production/client.

[ ] Set the credit cap and session timebox.

[ ] Write the no-go list.

[ ] Write the main user flow in plain language.

[ ] Define the smoke test.

During building

[ ] Change one major thing at a time.

[ ] After each major change, test the main flow manually.

[ ] Keep screenshots of working states.

[ ] Do not let polish features enter before the core loop works.

[ ] Track any cost spikes or AI credit drains.

[ ] Write down bugs as tickets instead of carrying them in your head.

After building

[ ] Run the smoke test.

[ ] Test on mobile.

[ ] Capture the demo flow in screenshots or screen recording.

[ ] Write the post-build review.

[ ] Decide: continue, pause, kill, sell, or reuse the architecture.
