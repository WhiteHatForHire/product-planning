Marcus’s Factory Machinery to Add

Core realization

I am not discovering this field from the outside. I am already on the path.

The external research mostly confirms patterns I have already reached through use: spec first, agent lanes, tests, deployment proof, logs, build ledgers, receipts, and human judgment over model magic.

What is different about my method is the archive layer.

A lot of AI-native builders are focused on shipping workflows, automations, apps, or content. My system is also building a long-term record: threads, build logs, cost estimates, compression ratios, journal entries, field notes, product specs, essays, and receipts.

That archive is not just memory. It is future machinery.

Capture raw material now. Build the factory next.

What makes my method different

1. Religious build logs

Every serious build should leave a record:

What was built.
 How long it took.
 What agents were used.
 What broke.
 What fixed it.
 What shipped.
 What would have taken a traditional team days or weeks.
 What the old agency cost would have been.
 What the AI-native cost actually was.
 What the next process improvement is.

This is not just journaling. This is operational evidence.

Factory Machinery to Add

1. Playwright as agentic QA worker

This is one of the clearest next upgrades.

I should treat Playwright not just as a testing library, but as an agentic QA worker.

What this means

The agent can open the app.
 Click through routes.
 Log in with a test account.
 Check layout at mobile/tablet/desktop.
 Capture screenshots.
 Inspect the accessibility tree.
 Confirm buttons exist.
 Confirm pages do not crash.
 Confirm forms submit.
 Confirm frontend talks to backend.
 Confirm error states are readable.
 Confirm production is actually usable.

Why it matters

This directly solves Anchor-type problems.

The app can “look fine” in code and still break in production. Playwright gives the agent eyes, hands, and a repeatable proof loop.

Add to factory

Create a reusable Agentic QA Playwright Runbook:

Test routes.
 Test auth.
 Test forms.
 Test mobile layout.
 Test desktop layout.
 Capture screenshots.
 Report failures with file paths and likely causes.
 Never merge without visual/smoke proof.

2. Strict TDD and context-reset loops

I already use tests and smoke checks, but the next level is stricter.

Pattern to test

One issue.
 One failing test.
 Implementation.
 Pass test.
 Commit.
 Flush context.
 Next issue.

This prevents agents from getting polluted by long, messy context. It also keeps the build from turning into a giant wandering patch session.

Why it matters

Agents get worse when context gets muddy. They start carrying wrong assumptions, half-fixed ideas, stale files, and previous false diagnoses.

A context reset after each verified slice keeps the agent clean.

Add to factory

Create a reusable TDD Slice Prompt:

Read only the issue.
 Write or identify the failing test.
 Do not implement until the test fails for the expected reason.
 Implement the smallest fix.
 Run tests.
 Commit if green.
 Summarize exactly what changed.
 Stop.

3. Skills and rules directories

This is probably the biggest practical upgrade.

Right now a lot of my taste lives in live prompting. That works, but it is not yet fully machine-loadable law.

The next step is to turn recurring standards into reusable files.

Skills/rules to create

PRD rules
 How I want product specs written.

Agent lane rules
 How to divide work across agents without overlap.

Deployment pre-flight rules
 API keys, env vars, directories, health endpoints, CORS, frontend-backend connection.

Visual QA rules
 How screenshots should be captured and reviewed.

Smoke test rules
 What “production ready” actually means.

Refactor rules
 When to refactor, when to patch, when to stop.

Git rules
 Branch naming, commits, PR hygiene, no dangerous commands without confirmation.

Done criteria
 What proof is required before saying “done.”

Cost ledger rules
 How to estimate pre-AI cost, AI-native cost, compression ratio, and margin.

Journal/build archive rules
 How to preserve the human/operator story alongside the technical record.

Add to factory

Create a /skills or /rules directory for each serious project, plus a global “Marcus Build OS” version.

This should become the private operating system for the way I build.

Not a public course yet. Not something casually handed to Keanan or anyone else. This is protected workflow IP.

Contents

1. Project intake template

2. PRD template

3. Agent lane map
